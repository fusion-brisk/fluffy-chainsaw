# Contentify — CJM для аудита интерфейса (Claude Design)

> **Назначение документа:** передать Claude Design (или внешнему дизайнеру) полную картину продукта и пользовательских путей для аудита интерфейса плагина.
>
> **Дата:** 2026-04-26
> **Версия плагина:** 3.1.2
> **Состояние документа:** актуально (после миграции на YC cloud-relay + heads-up narrative)

---

## 1. О продукте

**Contentify** — Figma-плагин, который автозаполняет макеты данными из живой поисковой выдачи Яндекса.

**Ценностное предложение для дизайнера:** вместо ручного копирования сниппетов (заголовков, цен, бейджей, картинок товаров) из реального SERP в макет — один клик на иконку расширения в Chrome → данные сами попадают в Figma-инстансы компонентов из дизайн-системы.

**Целевая аудитория** — продуктовые дизайнеры в Яндексе, которые:

- работают над поисковой выдачей (SERP) и ленточным фидом ya.ru
- рисуют макеты сниппетов в Figma и хотят видеть их с реальными данными вместо placeholder'ов
- проектируют под несколько брейкпоинтов (desktop, touch) и нужны разные варианты данных

### Что плагин умеет (на верхнем уровне)

1. **Импорт SERP** — парсит выдачу Яндекса (товары, магазины, органика, виджеты), заполняет соответствующие компоненты в Figma
2. **Импорт фида ya.ru** — парсит rhythm-feed (бесконечный поток карточек), заполняет специальные feed-компоненты
3. **Page Builder / Breakpoint Skeletons** — собирает целую страницу из импортированных сниппетов в нужном порядке для desktop/touch
4. **Component Inspector** — показывает свойства Figma-компонента, помогает проверить, правильно ли работает schema engine
5. **HTML Export** — экспортирует Figma-фрейм в HTML+CSS для передачи фронтенд-разработчикам
6. **Логи** — окно с DEBUG/SUMMARY-логами импорта (диагностика)
7. **What's New** — диалог с новинками после обновления версии

### Технический контекст (минимум, нужный дизайнеру)

```
Chrome Extension          YC Cloud Function           Figma Plugin (iframe)
─────────────────         ──────────────────          ─────────────────────
Парсит DOM SERP    →      POST /push                  GET /status (poll 1s)
Грузит JSON        →      UPSERT в YDB        →       Видит hasData / headsUp
                                                      → диалог подтверждения
                                                      → applies в Figma API
```

- **Окно плагина** — iframe в Figma шириной/высотой, которые плагин динамически меняет (resize)
- **Sandbox-ограничение** — внутренний код в Figma compileится в ES5, но это для UI неважно
- **Pairing** — при первом запуске генерируется 6-значный session code; расширение подхватывает его через URL-параметр на ya.ru (auto-pair)

## 2. Персоны

### Persona A — «Опытный дизайнер SERP» (90% времени работы)

- Пользуется плагином 5-15 раз в день, batch'ами по 3-5 импортов подряд
- Знает плагин наизусть, ожидает скорости и тишины (минимум кликов, минимум диалогов)
- Часто работает на двух мониторах: Figma слева, Chrome с Яндексом справа
- Болевые точки: задержка > 1 секунды, лишний клик, окно плагина перекрывает важные слои
- Что обожает: confetti при первом успешном импорте дня, точный narrative прогресса

### Persona B — «Новый пользователь / стажёр» (первый запуск)

- Первый раз открывает плагин, не понимает зачем код-связка
- Боится сломать что-то в Figma; не знает, какие фреймы выделить
- Болевые точки: непонятная иконка расширения, расплывчатые ошибки
- Что обожает: ясный onboarding wizard с пошаговыми инструкциями

### Persona C — «Возвращающийся после паузы» (раз в неделю)

- Плагин с прошлого раза мог обновиться, расширение могло отвалиться
- Сессия могла истечь (но в нашем случае session code хранится в clientStorage Figma — не истекает)
- Болевые точки: «всё было — теперь не работает», непонятно что чинить

## 3. UI-инвентарь (что вообще есть на экране плагина)

### Размеры окна (UI_SIZES)

| Tier     | Размер       | Когда используется                                                     |
| -------- | ------------ | ---------------------------------------------------------------------- |
| compact  | 320 × 56 px  | `ready` / `checking` / `incoming` / `processing` / `success` / `error` |
| standard | 320 × 280 px | `confirming` (диалог подтверждения)                                    |
| extended | 420 × 520 px | `setup` / панели (logs / inspector / settings / what's new)            |

**Внимание дизайнеру:** Figma добавляет ~40px заголовка ОТ окна iframe. Реальная высота контента = `height - 40px`. Это часто забывают, и окно «не помещается» в свой же tier.

### Компоненты (источник: `packages/plugin/src/ui/components/`)

| Компонент                | Где появляется                              | Назначение                                                  |
| ------------------------ | ------------------------------------------- | ----------------------------------------------------------- |
| `SetupSplash`            | первый кадр перед setup                     | плейсхолдер пока async-чтение clientStorage не вернулось    |
| `SetupFlow`              | onboarding (шаги: extension → pair → ready) | wizard для первого запуска                                  |
| `CompactStrip`           | 56-px полоска                               | unified компактное состояние с status icon + текст + меню   |
| `ImportConfirmDialog`    | 320×280 диалог                              | подтверждение импорта: query, count, source, summary fields |
| `Confetti`               | overlay при первом успехе                   | праздничная анимация при первом успешном импорте дня        |
| `OnboardingTip`          | thin banner над strip                       | one-time hint после онбординга: «нажми иконку в Chrome…»    |
| `UpdateBanner`           | thin banner над strip                       | предложение обновить расширение                             |
| `PairedBanner`           | thin banner над strip                       | flash-уведомление при успешном pairing                      |
| `CloudUnreachableBanner` | thin banner над strip                       | «облако недоступно, проверьте интернет»                     |
| `LogViewer`              | extended-панель                             | DEBUG/SUMMARY-логи импорта                                  |
| `ComponentInspector`     | extended-панель                             | свойства выделенного Figma-компонента                       |
| `WhatsNewContent`        | extended-панель после version bump          | список новинок этого релиза                                 |

### AppState FSM (текущая)

```
setup ──► checking ──► ready ──► incoming ──► confirming ──► processing ──► success ──► ready
   │          │          │           │             │              │              │
   ▼          ▼          ▼           ▼             ▼              ▼              ▼
 (init)   (5-8с probe) (idle 1с    (heads-up    (320×280 dialog)(N/M progress) (3с auto-
                       active poll) narrative)                                  dismiss)
                          │           │             │              │
                          └─ relay ───┘             └─ cancel ─────┴── error ──► error (5с auto-dismiss)
                             офлайн → ready
                             с offline-баннером
```

**Новое в этой ветке (только что мерджнули):** состояние `incoming` — компактный 56-px вид с narrative («Расширение собирает данные…», «Грузим скриншоты 7/27…», «Завершаем загрузку…») + ссылка «Отменить».

### Меню (доступно из CompactStrip → ⋮)

1. Логи
2. Инспектор
3. Настройки (опять открывает SetupFlow)
4. Что нового
5. Макеты брейкпоинтов
6. Экспортировать HTML (если есть выделение)
7. **Danger zone** (отделено сепаратором):
   - Очистить очередь (если есть pending)
   - Сбросить сниппеты

## 4. Customer Journey Maps

### CJM-1 — Первый запуск (Onboarding) — **Pivotal Journey**

> Если этот путь сломался, пользователь потерян. Каждый drop-off здесь = -1 потенциальный регулярный юзер.

| #   | Стадия               | Touchpoint                     | Что видит / делает                                                            | Эмоция              | Friction / возможные провалы                                                                             |
| --- | -------------------- | ------------------------------ | ----------------------------------------------------------------------------- | ------------------- | -------------------------------------------------------------------------------------------------------- |
| 1   | Install plugin       | Figma Community / link         | Кликает «Install», плагин в списке                                            | 🟢 интерес          | —                                                                                                        |
| 2   | First Launch         | Plugins → Contentify           | Видит `SetupSplash` → `SetupFlow` extended-окно (420×520)                     | 🟡 ожидание         | Если clientStorage медленный — splash-flash 100-300мс                                                    |
| 3   | Step 1: Install ext. | SetupFlow                      | Инструкция «установите Chrome extension», ссылка на CRX                       | 🟡 непонимание      | Самый страшный drop-off — почему мне нужно расширение? Объясняем ли мы это до того, как просим?          |
| 4   | Pair extension       | Кнопка «Связать»               | Плагин открывает `ya.ru/?contentify_pair=<6char>`. Расширение ловит, шлёт ack | 🟡 неопределённость | Если расширение не установлено — Yandex просто откроется без эффекта. Что плагин показывает в это время? |
| 5   | Pair-ack received    | `PairedBanner` (3с flash)      | Тонкая полоска «✓ Расширение связано»                                         | 🟢 облегчение       | Заметит ли пользователь полоску за 3 секунды?                                                            |
| 6   | Ready idle           | CompactStrip ready (320×56)    | Зелёная точка + «Подключено»                                                  | 🟢 готов            | Что делать дальше — непонятно. Тут нужен `OnboardingTip`!                                                |
| 7   | OnboardingTip        | Thin banner над strip          | «Откройте поисковую выдачу Яндекса и нажмите на иконку расширения»            | 🟢 ага              | Появляется только в первый ready-state. Что если пользователь свернул окно — увидит?                     |
| 8   | First import         | Click toolbar icon → narrative | (см. CJM-2 ниже)                                                              | 🟢 wow              | Главный момент истины. Если что-то здесь сломалось — пользователь потерян.                               |
| 9   | First success        | `Confetti` + success-state     | Конфетти + «N элементов · X.X сек»                                            | 🟢🟢🟢 восторг      | Конфетти случается ровно один раз — first-run flag persisted в clientStorage                             |

**Известные шероховатости (для аудита):**

- Шаг 2: переход `SetupSplash → SetupFlow` может flicker'ить (спека по `setup-skipped-loaded` уже частично починена, но визуально хочется проверить)
- Шаг 3: текст инструкции про расширение на extended-tier — не помещается без скролла на некоторых разрешениях (?)
- Шаг 5: 3-секундный flash может быть слишком быстрым для первого юзера (он только что кликнул кнопку, ещё смотрит в Chrome)
- Шаг 6: gap между `ready` и `OnboardingTip` — есть ли визуальная связь?

### CJM-2 — Ежедневный импорт (Core Loop) — **Repeating Journey**

> 5-15 раз в день. Каждая лишняя секунда × 10 повторений = ощутимая потеря продуктивности.

| #   | Стадия            | Touchpoint                       | Что видит / делает                                                                   | Эмоция          | Friction                                                                                          |
| --- | ----------------- | -------------------------------- | ------------------------------------------------------------------------------------ | --------------- | ------------------------------------------------------------------------------------------------- |
| 1   | Resume work       | Открыл Figma + плагин            | Plugin вспоминает session, переходит `setup → checking → ready` за 5-8с              | 🟡 ожидание     | 5-8с — это много. Cold-start YC API Gateway виноват                                               |
| 2   | Browse SERP       | Chrome → yandex.ru/search?text=… | Открыл выдачу (часто на втором мониторе)                                             | 🟢 в потоке     | —                                                                                                 |
| 3   | Trigger import    | Click extension icon             | Иконка badge меняется на «...»                                                       | 🟢 лёгкость     | Если на не-Yandex странице — иконка серая, badge ✗, уведомления нет                               |
| 4   | **NEW: Incoming** | CompactStrip incoming (56px)     | «Расширение собирает данные…» → «Грузим скриншоты 7/27…» → «Завершаем загрузку…»     | 🟢 «работает!»  | Раньше тут было idle 1-5с. Теперь narrative — но фразы должны быть читаемы за 1с                  |
| 5   | Cancel (rare)     | «Отменить» link                  | Плагин в `ready`, очередь очищена                                                    | 🟡 контроль     | Кнопка маленькая, легко промахнуться?                                                             |
| 6   | Confirm dialog    | ImportConfirmDialog (320×280)    | Query, count, source, summary fields, «В выделение» / «На страницу», «Импортировать» | 🟢 контроль     | Если пользователь работает быстро — каждый раз кликать «Импортировать» утомительно. Auto-confirm? |
| 7   | Processing        | CompactStrip processing (56px)   | Spinner + «Размещаем 23/76…» + 3-px progress bar внизу                               | 🟡 ожидание     | Sandbox-фаза = 5-30с в зависимости от объёма                                                      |
| 8   | Success           | CompactStrip success (56px)      | Зелёная галочка + «76 элементов · 12.3 сек», 3-px filling progress (3с auto-dismiss) | 🟢 завершение   | 3 секунды auto-dismiss — иногда хочется задержать, чтобы посмотреть статистику                    |
| 9   | Back to ready     | CompactStrip ready               | Возврат в idle                                                                       | 🟢 готов к next | —                                                                                                 |

**Острые места для аудита:**

- Шаг 1: 5-8с холодного старта YC — есть ли визуальная индикация что «прогреваем», или просто spinner?
- Шаг 4: новый narrative — ВСЕ ФРАЗЫ помещаются в 56-px полоску? Не обрезаются на длинных значениях K/M (например 99/127)?
- Шаг 5: «Отменить» не выделено цветом — visible enough?
- Шаг 6: `ImportConfirmDialog` имеет radio-кнопки «В выделение» / «На страницу». Дизайн не пересматривался с момента редизайна (Mar 2026). Стоит свежим взглядом.
- Шаг 7: progress bar 3px — слишком тонко? Виден на retina?
- Шаг 8: confetti **не** появляется на 2-м, 3-м импорте — это правильное решение или teach-and-fade проблема?

### CJM-3 — Repair / re-pair после паузы

> Пользователь возвращается через неделю. Что-то могло сломаться.

| #   | Стадия               | Touchpoint               | Что видит / делает                                            | Эмоция          | Friction                                                                |
| --- | -------------------- | ------------------------ | ------------------------------------------------------------- | --------------- | ----------------------------------------------------------------------- |
| 1   | Open plugin          | Plugins → Contentify     | Plugin грузится, видит сохранённый session, идёт в `checking` | 🟢 ожидание     | Если YC обновился, cold-start ~5с                                       |
| 2   | Cloud unreachable    | `CloudUnreachableBanner` | Баннер «Не удалось подключиться к облаку, проверьте интернет» | 🔴 расстройство | Что делать? Кнопка «Повторить» есть, но нет диагностики причины         |
| 3   | Retry succeeded      | Banner исчезает          | Возврат в `ready`                                             | 🟢 облегчение   | —                                                                       |
| 4   | Extension недоступно | (не очевидно)            | Plugin в `ready`, но иконка расширения серая на Yandex        | 🟡 confused     | Plugin не знает, что extension сломалось — он просто не получает данных |
| 5   | Repair via menu      | ⋮ → «Настройки»          | `SetupFlow` открывается в repair-mode (allowRepair=true)      | 🟡 надежда      | Repair-mode стартует со step pair — повторное связывание                |
| 6   | Fresh pair           | Кнопка «Связать»         | Новый pair handshake, отправляется новый ack                  | 🟢 ремонт       | —                                                                       |

**Острые места:**

- Шаг 2: Banner не указывает причину (offline, CORS, cold-start). Полезно ли разделить «облако не отвечает» и «нет интернета»?
- Шаг 4: «Иконка серая» — это сигнал из Chrome, плагин его не знает. Можем ли мы как-то догадаться, что extension установлено vs не установлено?
- Шаг 5: вход в repair только через меню — не очевидно. Может, баннер «Перепарить» вместо CloudUnreachable, если pair-ack не приходит >X секунд?

## 5. Узкие места и open questions для дизайн-аудита

### Hot zones (где, по нашему мнению, нужны улучшения)

1. **OnboardingTip vs SetupFlow** — после онбординга tip появляется один раз. Достаточно ли этого? Не теряется ли он на фоне CompactStrip?
2. **CompactStrip 56px высота** — текстовые сообщения могут быть длинными («Расширение собирает данные…», «Размещаем 127/543…»), помещается ли всё с учётом 8px padding?
3. **Меню кнопка ⋮** — единственный путь ко всем второстепенным фичам (logs, inspector, settings, page-builder, html-export). Нет ли когнитивной перегрузки?
4. **Danger zone** — «Очистить очередь» и «Сбросить сниппеты» отделены сепаратором, но в том же выпадающем меню. Достаточно защиты от случайного клика?
5. **Confirm dialog (320×280)** — узкий, контент может скроллиться при длинном query. Стоит проверить.
6. **CloudUnreachableBanner** — не различает «нет интернета», «CORS-ошибка», «cold-start». Стоит ли быть точнее?
7. **Auto-dismiss success (3с) и error (5с)** — хватает ли времени дочитать?

### UI consistency

- Тёмная / светлая темы Figma — все ли цветовые переменные имеют hardcoded fallback'и?
- Touch (mobile / iPad) платформа — usePlatform делает overlay для меню как bottom sheet. Тестировался ли иконометрический набор?
- Шрифты — используем системный шрифт Figma. Везде ли консистентно?

### Accessibility (мы поверхностно прошлись, нужен системный аудит)

- ARIA-метки на всех кнопках есть (после Apr 2026 фикса)
- Focus trap в модалях есть
- Arrow key navigation в меню есть
- НЕ проверялось: контрастность всех элементов, screen-reader flow для онбординга, keyboard-only flow для импорта

### Новое (не аудитили) — добавлено в этой неделе

- AppState `'incoming'` + narrative phrases (см. CJM-2 шаг 4)
- Cancel link в `incoming` mode CompactStrip (стиль скопирован с других action-link'ов)
- Spinner variant `--incoming` (тот же что у processing, но с brand-цветом)

## 6. Что мы хотим от Claude Design

### По убыванию приоритета

1. **Ревью CJM-2 (core loop)** — мы делаем 5-15 импортов в день. Каждая секунда / лишний клик / неточная фраза умножается. Критично оценить новый `incoming` narrative.
2. **Ревью CJM-1 (onboarding)** — pivotal journey. Дополнительные drop-off'ы здесь = потерянные пользователи. Особенно интересует шаг 4 (pair → ack → flash) — visible ли успех?
3. **Ревью CJM-3 (repair)** — наименее проработан. Когда что-то ломается, пользователь часто не знает что чинить. Какие сигналы добавить?
4. **Глобальные UI consistency и accessibility** — шрифты, цвета, контрастность, screen-reader.
5. **Опциональный bonus** — стоит ли пересмотреть форму меню ⋮ → выпадающий список → 8 пунктов? Или это норма?

### Формат ожидаемой обратной связи

- Список конкретных проблем с серьёзностью (Critical / Important / Minor)
- Ссылки на конкретные компоненты и состояния FSM (см. секцию UI Inventory)
- Предложения с альтернативами, не требующими полного редизайна

### Что НЕ нужно (out of scope)

- Брендинг и логотипы (уже зафиксированы)
- Глубокая ревизия Figma-design-system (отдельный продукт)
- Производительность — мы её мониторим сами
- Архитектура backend — про cloud-relay и YDB не нужно
